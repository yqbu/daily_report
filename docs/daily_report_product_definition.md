### 一句话定位

> daily_report 是一个只为产品所有者本人服务的个人每日数字活动回顾工具：它在不要求持续手工记录的情况下，把当天分散的数字行为线索整理为经过用户确认的事实回顾，帮助用户解释主要时间去了哪里。

### 一句话使命

> 让碎片化的一天不再只能依赖模糊记忆来解释。

### 一句话核心价值

> 以可接受的回顾成本，将零散、私有的行为痕迹转化为可信且有意义的当日事实，使用户重新获得对自己时间投入的理解权和决策权。

### 产品类别

**个人每日数字活动回顾工具**，而不是传统的工作日报生成器或精确时间追踪器。

### 如果只能做好一件事

> 帮助用户真实、概括性地解释当天主要时间投入到了哪些事情上。

### 产品不是什么

- 不是单纯的行为数据采集器
- 不是分钟级工时核算工具
- 不是任务或项目管理工具
- 不是效率评分或行为纠正工具
- 不是员工监控或团队管理工具
- 不是长期知识管理系统
- 不是个人原始数据中心
- 不是保存所有 AI 对话和网页内容的归档工具
- 不是脱离私有上下文的一次性 AI 写作工具

## 🎯 产品存在理由

### 根本问题

**事实：** 用户的普通工作日通常同时包含两到三个并行项目、历史项目修复、临时任务、文档或 PPT 工作，以及工作间隙中的技术阅读和探索。频繁切换、零碎事项与临时打断使这些活动难以形成清晰记忆。

当用户在下班前尝试回顾时，常出现以下失败链条：

> 需要回顾当天 → 记忆不足 → 翻找浏览记录或聊天记录 → 线索分散且解释成本过高 → 放弃回顾。

### 第一优先级问题

当一天由大量碎片化数字活动组成时，用户缺少一份能够直接说明“为了什么事项，做了什么”的事实记录，因而无法低成本地解释主要时间去向。

### 第二优先级问题

用户偶然接触的技术方案、学习内容和参考页面容易失去上下文。daily_report 可以在当日报告中保留少量经确认的结论、页面标题和链接，为后续追溯留下线索，但不承担长期知识管理职责。

### 问题强度

- **事实：** 除高度忙碌且目标单一明确的日子外，“时间去了哪里”的不确定感几乎每天都会出现
- **事实：** 高度专注于单一事项时，用户通常能够清楚回忆当天活动，产品的增量价值较低
- **事实：** 问题最强的场景是多项目并行、零碎事项密集、临时打断频繁的普通工作日
- **后果：** 用户感到焦虑和失控，可能放弃回顾；填写项目人时时需要询问同事；找不到历史技术线索时可能重复构思

### 现有替代方式为什么不够

| 替代方式 | 可以解决什么 | 主要缺口 |
| --- | --- | --- |
| **凭记忆回顾** | 快速想起显著事项 | 碎片活动容易遗漏，无法解释模糊时间段 |
| **查看浏览历史** | 找回部分网页访问 | 需要人工翻找，无法说明事项背景和实际目的 |
| **查看聊天记录** | 找回沟通和部分任务线索 | 只覆盖沟通行为，仍需人工拼接 |
| **手工工作日志** | 形成结构化记录 | 要求白天持续维护，增加一套新的记录负担 |
| **任务管理工具** | 说明计划做什么 | 不能说明当天实际发生了什么 |
| **普通时间追踪工具** | 说明应用或网站用了多久 | 不能说明“为了什么事项，做了什么” |
| **直接使用通用 AI** | 可以完成大部分归纳总结 | 不天然具备用户可控的材料边界和持续存在的个人项目背景 |
| **不进行回顾** | 不产生即时成本 | 持续承受时间不知去向、历史事实难以重建的成本 |

### 产品不存在时的成本

- 每日回顾常因想不起来或找不到线索而放弃
- 用户难以判断主要时间是否投入到自己关心的方向
- 项目人时回顾需要依赖同事或零散历史记录
- 值得保留的学习与探索线索容易失去上下文
- 用户需要重复翻找、拼接，甚至重新构思曾经接触过的方案

## 👤 目标用户与核心任务

### 核心目标用户

当前唯一目标用户是产品所有者本人。产品不为通用性、市场规模或其他相似用户优化。

目标用户具备以下工作特征：

- 同时推进多个开发项目或工作事项
- 经常处理历史项目修复和临时任务
- 大部分活动发生在电脑和浏览器中
- 工作中存在频繁切换、打断和零碎事项
- 会在工作间隙阅读技术博客、方案和学习材料
- 不愿在白天事无巨细地维护另一套工作日志
- 愿意在下班前集中投入十几分钟到半小时进行有价值的回顾

### 次要用户

当前没有次要用户。主管、项目成员和组织不是产品服务对象。用户可以将确认后的日报用于工作汇报，但产品不为汇报对象设计。

### 不适合的用户

- 需要团队工时核算、审批或绩效评价的人
- 需要分钟级计费或合规工时记录的人
- 希望产品主动限制娱乐、监督专注或评价效率的人
- 希望统一管理任务、项目、知识和个人数据的人
- 工作几乎不产生数字行为痕迹的人

### 核心 Job to Be Done

> 当一天被多个项目、临时事项和探索活动切碎时，我希望无需依赖模糊记忆或反复翻找，就能获得一份可信的当日事实回顾，从而理解自己的主要时间投入到了哪里。

### 相关任务

- 当需要填写项目人时时，按天查看历史回顾以辅助恢复记忆
- 当偶然想回看当天此前某段时间时，查看已经形成的活动概括
- 当回忆起某项技术阅读时，通过日报中保留的标题和链接追溯来源
- 将长期保存的每日报告交给其他工具进行跨天整理和分析

### 不属于产品的任务

- 决定明天应该做什么
- 维护项目待办和未完成事项
- 自动判断哪些时间值得或浪费
- 进行跨周、跨月的长期规律分析
- 建立完整的个人知识库
- 保存所有原始行为、网页全文或 AI 完整回答

## 🔄 用户场景与核心闭环

### 核心场景：每日回顾

每天下班前十几分钟，当主要工作已经结束、节奏开始放缓时，用户打开 daily_report。产品已经依据当天行为线索形成待审材料，但任何可能离开用户控制范围的内容都必须先通过规则过滤与用户检查。

用户检查十几至二十几条无法确定或风险较高的材料，排除敏感内容，必要时补充线下事项、工作成果或个人感悟。人工补充是可选行为，没有内容需要补充时可以直接继续。

产品随后生成事实性日报草稿。草稿中的大部分事项、内容和项目归属应当已经正确。用户进行确认、修正归属和少量补充；只有经过确认的日报才成为长期记录。

用户在能够大致回答以下问题时结束回顾：

- 今天主要做了哪些事情
- 这些事情与哪些项目或投入方向有关
- 今天看了什么、理解或学到了什么
- 每个整合后的事项大致占用了多少时间
- 当天主要时间大致分布在哪些应用中

### 次要场景

| 场景 | 触发时刻 | 产品责任 |
| --- | --- | --- |
| **偶然回顾** | 临时想知道当天此前做过什么 | 查看当天已经形成或正在形成的事实材料 |
| **每周查阅** | 周五回顾或填写项目人时 | 按天查看已确认日报，不生成独立周总结 |
| **历史追溯** | 想起某天的事项或参考页面 | 按日期查看日报及其中保留的少量证据 |

### 用户为什么持续回来

- 碎片化和频繁切换几乎每天都会重新发生
- 用户希望避免一天结束后无法解释时间去向
- 经确认的日报会形成可信、可长期回看的个人事实记录
- 每日记录可以辅助后续项目人时回顾和外部工具的长期整理

## 💡 核心价值与结果标准

### 用户结果、产品能力与支撑能力

| 层级 | 定义 | daily_report 中的体现 |
| --- | --- | --- |
| **用户结果** | 用户最终获得的改变 | 能够解释当天主要时间去了哪里 |
| **核心能力** | 直接产生用户结果的能力 | 将零散痕迹归纳为“为了什么事项，做了什么” |
| **支撑能力** | 保证核心能力可靠成立 | 线索采集、隐私过滤、项目背景、状态确认、临时证据保存 |

### 核心增量价值

daily_report 的不可替代价值不是“把记录放在一起”，而是完成以下转化：

> 分散且不愿逐条阅读的行为痕迹 → 有上下文、有事项含义、可由用户确认的每日事实。

它比普通时间追踪多解释了活动目的，比任务管理多解释了实际发生的事情，比手工日志减少了持续记录负担，比通用 AI 多提供了私有材料控制与持久个人背景。

### 每日报告的长期成果

经过用户确认的正式日报可以包含：

- 当天主要事项的事实概括
- 事项与项目或投入方向的关联
- 每个事项的概括性时间投入
- 经确认的学习、探索或问题结论
- 与当天事项有关的少量页面标题和链接
- 各应用的大致使用时长
- 用户主动补充的线下事项、成果或感悟

### 真实性标准

- 真实性优先于覆盖率
- 可以遗漏部分事项，但不能写入没有发生的事项
- 项目归属错误可以由用户修正
- 无法解释的时间可以留白，不需要强行补齐
- 页面访问只能证明发生过浏览，不能自动证明已经理解或采纳
- AI 提问只能证明发生过询问或探索，不能自动证明问题已经解决
- 未经用户确认的草稿不是长期可信记录

## 🚫 产品边界

### 必须负责

- 在不要求持续手工记录的前提下获得当天数字行为线索
- 在用户允许的材料范围内形成事实性日报草稿
- 将零散痕迹归纳为有意义的事项，而不是展示原始流水
- 让用户在生成前控制敏感材料边界
- 让用户确认和修正日报后再长期保存
- 允许按日期查看历史日报
- 对未采集、证据不足和无法确认的部分保持诚实

### 可以辅助

- 通过每日事实帮助填写项目人时
- 保留少量页面标题和链接以便追溯
- 保存应用使用时长作为辅助统计
- 将已确认的日报提供给其他工具继续使用
- 通过项目背景提高事项归纳准确性

### 当前不做

- 跨日报搜索、关联和长期趋势分析
- 独立的周报、月报和长期总结
- 完整 AI 回答归档
- 网页全文长期保存
- 原始行为数据永久保存
- 未验证的复杂应用分类体系

### 永远不应做

- 评价用户是否努力、自律或高效
- 将休息和探索自动判定为浪费时间
- 对用户进行员工监控或绩效评分
- 为追求完整而虚构事项、成果或时间
- 在用户确认前使用明确敏感或高风险材料
- 强迫用户每天填写补充内容才能完成日报

### 应交给其他工具

| 产品类型 | 应承担的职责 |
| --- | --- |
| **任务管理** | 管理准备做什么、待办和优先级 |
| **项目管理** | 管理计划、里程碑、进度和协作 |
| **知识管理** | 跨天整理、关联、搜索和沉淀知识 |
| **精确时间追踪** | 分钟级计时、计费与正式工时核算 |
| **行为分析** | 长期趋势和模式分析，但不得成为 daily_report 的核心 |
| **团队汇报** | 面向主管、项目成员或组织的标准化汇报 |
| **AI 对话平台** | 保存完整 AI 对话和回答 |
| **个人数据中心** | 为未知未来用途长期维护完整原始数据 |

### 产品责任分界

- 项目管理工具描述“准备做什么”；daily_report 解释“今天实际做了什么”
- 时间追踪工具描述“哪个应用用了多久”；daily_report 解释“这段时间为了什么事项做了什么”
- 知识管理工具负责跨天整理；daily_report 对每一天的事实回顾负责
- AI 写作工具生成文字；daily_report 还必须为材料边界、个人背景和用户确认负责

## 🔍 现有能力处置

以下处置依据本次 Product Discovery 中确认的用户价值，不以开发成本和完成程度作为保留理由。

| 现有能力 | 处置 | 用户价值依据 |
| --- | --- | --- |
| **汇总一天的工作活动** | 保留为核心 | 直接形成能够解释时间去向的事项 |
| **生成结构化日报** | 保留为核心 | 将事实材料转化为可确认、可长期保存的每日成果 |
| **查看与筛选工作材料** | 保留为核心 | 在任何可能的外部处理前建立隐私边界 |
| **用户确认和修正日报** | 保留为核心 | 真实性与长期记录可信度的最终门槛 |
| **应用使用记录** | 降级为支撑 | 辅助估算时间段和应用分布，不能单独解释事项 |
| **浏览器访问与浏览行为** | 降级为必要支撑 | 页面级线索能够说明调研主题、项目背景并支持追溯 |
| **记录用户向 AI 提出的问题** | 降级为必要支撑 | 经常能够还原当时为哪个事项处理了什么问题 |
| **提炼 AI 对话结论** | 需要验证 | 有助于记录所得结论，但必须避免把询问误写为解决 |
| **完整保存 AI 回答** | 建议移除 | 日报只需要问题与经确认的结论，完整归档应交给 AI 平台或知识管理工具 |
| **剪贴板工作线索采集** | 需要验证，倾向移除或默认排除 | 当前理由仅是提高覆盖率，尚无实际价值证据，隐私风险较高 |
| **人工补充工作材料** | 降级为必要支撑 | 补充线下事项、成果和感悟；完全可选，不作为每日强制输入 |
| **项目背景记录** | 降级为必要支撑 | 帮助把活动归入用户确认的项目，不承担待办和进度管理 |
| **应用采集允许／排除规则** | 降级为必要支撑 | 保护隐私、减少噪声并降低每日人工审查数量 |
| **应用分类体系** | 保留观察 | 当前未稳定改善日报，只有证明提高归纳准确性后才升级 |
| **敏感信息识别与排除** | 保留为必要支撑 | 隐私控制是产品成立前提，自动识别不能替代用户判断 |
| **采集状态查看** | 保留为必要支撑 | 帮助确认当天是否存在记录以及日报为何可能不完整 |
| **本地运行技术状态** | 降级为支撑 | 只保留影响采集可信度的状态，开发运维信息不进入核心体验 |
| **细粒度原始记录长期保存** | 降级为临时支撑 | 原始证据可保留有限周期，不建设永久个人数据中心 |
| **按日期查看日报** | 保留为核心成果入口 | 支持偶然回顾、每周查阅和历史事实追溯 |
| **跨天搜索、关联和分析** | 建议移除 | 更长周期能力交给 Obsidian、Agent 或其他知识管理工具 |
| **页面标题和链接长期保留** | 保留为可选增强 | 经确认的少量证据有助于未来追溯，不保存完整浏览历史 |
| **应用时长写入日报** | 保留为可选增强 | 以紧凑方式辅助理解数字活动时间分布 |

## 📌 产品原则与成功标准

### 产品原则

1. **解释时间去向，而不是记录一切。** 数据源和记录数量只有在提高每日回顾质量时才有价值
2. **真实性优先于覆盖率。** 宁可遗漏或留白，也不能制造没有发生的事项
3. **隐私筛选先于内容处理。** 高风险和不确定材料必须由用户决定是否参与总结
4. **呈现事实，不评价用户。** 产品提供自省依据，不判断时间是否值得、用户是否高效
5. **自动化负责整理，用户拥有最终判断权。** 未经用户确认的草稿不能成为长期记录
6. **不能要求用户维护另一套详细日志。** 白天持续手工记录会破坏产品降低负担的价值
7. **对每天负责，把长期分析交给其他工具。** 跨天知识管理和趋势分析不能反向主导定位
8. **支撑能力不能抢占核心体验。** 配置、状态、数据源和原始流水必须让位于材料筛选、事实归纳与用户确认

### 用户成功标准

- 用户能在十几分钟到半小时内完成一次有价值的每日回顾
- 日报能够概括上午、下午等主要时间段做过的事情
- 日报通常能够解释约 70%～80% 的主要时间去向
- 草稿中的大部分事项、内容和项目归属已经正确
- 用户主要进行确认和少量修正，而不是重建整份日报
- 日报不包含没有发生的事项
- 无法确认的时间和活动可以明确留白
- 每日待人工判断的高风险或不确定材料通常控制在十几至二十几条
- 用户不再需要重复翻找多个来源才能理解当天
- 用户愿意持续使用，并认为回顾结果足以缓解“时间去了哪里”的不确定感

### 不能代表产品价值的指标

- 采集了多少条原始行为
- 接入了多少数据源或 AI 平台
- 原始数据保存了多长时间
- 应用时长是否精确到分钟
- 日报生成了多少字
- 用户打开了多少配置和状态页面
- AI 调用了多少次或使用了哪个模型

## ⚠️ 未决问题、风险与后续门槛

### 尚未验证的关键假设

- 行为线索中包含足够上下文，可以稳定还原“为了什么事项，做了什么”
- 自动草稿可以做到大部分事项正确，同时避免虚构内容
- 预设允许／排除规则可以把每日人工隐私审查控制在可接受范围
- 约 70%～80% 的可信覆盖足以缓解用户的时间不确定感
- 用户愿意长期维持下班前的确认习惯
- 项目背景能够提高归纳准确性，而不会演变为项目管理

### 尚未决定的事项

- 细粒度原始行为记录最终保留一个月还是一个季度
- “大部分事项正确”在真实使用中对应怎样的修改量
- AI 对话结论提炼是否能够达到真实性要求
- 应用分类是否确实能够改善事项归纳
- 剪贴板线索是否存在足以覆盖隐私风险的增量价值

### 最容易造成产品膨胀的方向

- 为未来可能有用而永久保存所有原始数据
- 不断增加数据源，却没有提高日报可信度
- 从每日回顾扩张到长期知识管理和个人数据中心
- 从事实呈现扩张到效率评价、行为干预和生产力监督
- 从项目背景扩张到待办、进度和项目管理
- 从保留少量证据扩张到网页全文和完整 AI 对话归档

### 后续真实使用需要验证

- 一次完整回顾的实际耗时
- 草稿遗漏、错误归属和虚构事项的发生频率
- 用户每次需要修正或补充多少内容
- 日报实际能够解释多少主要时间
- 高风险待确认材料的实际数量
- 用户在连续使用一个月后的持续意愿和主观价值
