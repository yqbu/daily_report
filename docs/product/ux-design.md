### 一句话体验定义

> daily_report 是一个安静、可信、由用户掌控的每日回顾空间：用户先理解当天发生了什么，再用少量判断和修正确认一份真实的每日事实，然后带着清楚感离开。

### 核心体验结果

一次成功的 daily_report 使用应让用户能够回答：

- 今天主要做了哪些事情
- 每件主要事项大概投入了多少时间
- 时间主要分布在哪些应用与活动时段
- 哪些事项属于工作项目或其他投入方向
- 是否存在值得记录的结果、学习或未完成内容

产品不要求解释每一分钟。通常能够真实解释约 70% 至 80% 的主要时间，即可完成回顾。

### 设计总方向

> 以文档感为主、工具感为辅的安静个人回顾工作台。

体验围绕五个动作展开：

> 筛选材料 → 理解事实 → 修正错误 → 确认回顾 → 离开产品

## 1. User Story

### 1.1 用户与工作环境

当前唯一目标用户是产品所有者本人。用户的普通一天具有以下特征：

- 同时推进多个工作项目，例如 APS、WES、RCS
- 穿插历史问题修复、临时事项、会议、文档和资料调研
- 经常在多个应用、网页和交流工具之间切换
- 也可能发生学习、娱乐和其他个人活动
- 大部分活动会留下数字线索，但用户不愿白天持续手工记录
- 下班前能够回忆主要事件，却容易忘记零散活动和时间分布

### 1.2 主要使用故事：下班前回顾

下班前，用户已经不适合再投入一项新的重工作，开始产生“今天到底做了什么，时间去了哪里”的模糊感。此时用户打开 daily_report，希望在十几分钟到半小时内：

1. 快速看到今天已有的活动线索与时间轮廓
2. 排除敏感或不需要进入回顾的材料
3. 获得按事项组织的回顾草稿
4. 核对事项是否真实、项目归属是否正确、时间是否大致合理
5. 确认一份简洁的每日回顾
6. 带着清楚感结束一天

用户打开前的主要心理状态不是“我要认真填写一份报告”，而是：

> 今天差不多结束了，我想快速弄清楚主要时间花在哪里。

### 1.3 次要使用故事：工作中临时查看

用户偶尔会在工作中突然想知道“今天到现在做了些什么”。这时用户只希望：

- 查看最近活动线索
- 查看主要应用和活跃时段
- 找回已经忘记的零散活动
- 确认记录是否正常
- 随后继续工作

这个场景是被动查看，不应自动进入材料筛选或形成草稿。

### 1.4 第三使用故事：回看最近几天

用户偶尔查看前几天的已确认记录，希望快速恢复近期上下文：

- 最近主要在做哪些事项
- 哪些工作项目反复出现
- 某一天是否看过某个资料或处理过某个问题
- 填写项目人时时，帮助恢复按天记忆

用户需要的是按日阅读，不是周报、趋势分析或项目进度面板。

### 1.5 用户何时不需要 daily_report

- 白天正在专注工作，不希望被要求记录
- 当天目标单一明确，用户已经能够清楚回忆
- 用户不想为某一天创建回顾
- 用户只需要完成任务，而不是理解已经发生的事情

### 1.6 用户何时会放弃回顾

以下情况容易导致放弃：

- 必须逐条阅读全部原始材料
- 必须为每条材料分配事项或项目
- 必须填写完整的结果、学习和明日计划
- 草稿虚构内容太多，需要重写整份文档
- 页面充满统计、配置和内部信息，找不到下一步
- 产品把缺少记录解释成用户没有工作
- 确认流程过长，或每一步都出现二次确认

### 1.7 用户为什么持续回来

- 碎片化和频繁切换会每天重新发生
- 一份已确认记录能够缓解“时间去了哪里”的不确定感
- 历史回顾能够辅助后续记忆恢复
- 产品不要求白天维护另一套日志
- 用户始终能够决定什么材料被使用、什么事实被保存

### 1.8 User Story 设计理由与边界

设计将下班前回顾设为核心、临时查看设为被动路径，是因为用户的真实意愿是在一天结束时集中整理，而不是全天管理自己。它避免 daily_report 膨胀为任务管理、持续时间追踪或行为监督工具。

## 2. User Flow

### 2.1 总体体验结构

daily_report 包含三条彼此独立但相互衔接的路径：

**日常观察**

> 打开应用 → 今日总览 → 查看当前线索与状态 → 继续工作或离开

**主动回顾**

> 今日总览 → 开始今日回顾 → 材料筛选 → 回顾草稿 → 正式确认 → 阅读或离开

**历史回看**

> 历史回顾 → 选择日期 → 阅读已确认的每日回顾 → 按需查看活动线索 → 离开

### 2.2 默认打开流程

打开应用默认进入今日总览，不自动开始回顾，也不自动形成草稿。

今日总览首先帮助用户理解：

1. 截至当前有哪些可识别的活动线索
2. 时间主要出现在哪些应用和时段
3. 当前记录是否正常
4. 今日回顾处于未开始、进行中还是已确认

用户可以直接离开，不产生待办或未完成压力。

### 2.3 核心回顾流程

核心流程由三个用户可理解的阶段组成：

#### 阶段一：材料筛选

目标：确认本次回顾允许使用哪些材料。

流程：

1. 默认查看高风险与不确定材料
2. 对材料选择“用于本次回顾”“不用于本次回顾”“稍后判断”或“标记为敏感”
3. 尚未判断的材料默认不使用，不阻塞继续
4. 用户按需进入全部材料审查
5. 全量审查提供来源、时间、状态和关键词筛选
6. 用户主动选择“形成回顾草稿”

完成标准不是看完所有材料，而是用户接受当前材料边界。

#### 阶段二：回顾草稿

目标：核对今天主要做了什么。

默认顺序：

1. 查看主要事项
2. 核对事项是否真实
3. 修正工作项目归属
4. 校准粗略时间
5. 按需查看相关活动线索
6. 修正啰嗦或不准确的表达
7. 查看紧凑的应用与时间分布，用于辅助回忆
8. 可选补充遗漏事项

确认前只轻量询问一次：

> 今天还有没有值得记下、但没有出现在上面的一件事？

用户可以只填写事项名称，也可以跳过。

#### 阶段三：正式确认

目标：把经过核对的草稿转为可信的个人记录。

流程：

1. 用户选择“确认每日回顾”
2. 不出现额外二次确认窗口
3. 页面保持原位，从草稿状态转为已确认文档
4. 用户阅读后直接离开
5. 后续编辑会让文档恢复草稿身份，并需要重新确认

### 2.4 中途离开与跨天处理

- 当天离开后再次打开，直接续接原来的回顾阶段
- 今日总览显示“材料待判断”或“草稿待核对”等真实状态
- 跨天后默认进入今天，不强制先处理昨天
- 今日总览弱提示“昨天的每日回顾尚未确认”
- 用户可以继续昨天、暂时忽略，或选择“这一天不创建每日回顾”
- 不使用逾期、红点、连续提醒或完成率

### 2.5 工作中临时查看流程

> 打开今日总览 → 扫描最近活动、主要应用和活跃时段 → 按需展开一条线索 → 继续工作

临时查看不会形成草稿，不改变回顾状态，也不要求保存。

今日总览默认表达今天。如果保留现有日期查询能力：

- 日期查询降为次要入口
- 切换到过去日期或日期范围后，页面名称改为“活动概览”
- 非今日状态不显示“开始今日回顾”
- 返回今天后恢复“今日总览”身份

### 2.6 历史回看流程

> 进入历史回顾 → 默认打开最近确认日期 → 从日期列表扫描近期主要事项 → 阅读选中日期 → 按需查看线索或编辑

历史回顾只收录已确认记录。未确认草稿留在今日回顾流程中。

### 2.7 材料不足或记录缺失

材料不足时，产品不推测用户没有做事，也不强行补齐。

允许以下结果：

- 提醒“今天只有少量活动线索”
- 将无记录时段表达为“暂无活动线索”，而不是空闲或无效时间
- 由用户补充一件主要事项
- 时间、项目和结果保持为空
- 只有一句话或一个事项也可以确认成每日回顾
- 用户可以明确选择这一天不创建记录

### 2.8 严重运行异常流程

> 全局出现严重提示 → 用户选择“紧急停止全部记录” → 明确展示实际停止结果 → 保持停止或尝试恢复 → 出现新活动线索后才恢复为正常状态

紧急停止不需要二次确认，因为它用于防止问题继续扩大，并且不会删除已有材料和设置。

### 2.9 时间目标

| 场景 | 目标时长 |
| --- | --- |
| 工作中快速查看 | 30 秒至 2 分钟 |
| 正常每日回顾 | 8 至 20 分钟 |
| 需要全量材料审查 | 最多延长至约 30 分钟 |
| 历史回看 | 1 至 5 分钟 |
| 正常运行状态检查 | 十几秒 |

### 2.10 User Flow 设计理由与边界

三阶段流程对应用户真实心智：先控制材料边界，再核对事实，最后确认记录。它取代以“素材选择、Prompt 配置、内容确认、结果预览”为中心的形成过程描述，避免用户承担不必要的内容生成管理负担。

## 3. Information Architecture

### 3.1 导航层级

#### 每日核心

- 今日总览
- 今日回顾
- 历史回顾

#### 运行与查阅

- 数据中心
- 运行中心

#### 管理

- 应用配置
- 设置

这些入口可以保持一层可达。分组只表达使用频率和产品职责，不要求隐藏现有模块。

### 3.2 模块职责

| 模块 | 核心职责 | 默认不承担 |
| --- | --- | --- |
| 今日总览 | 被动理解今天的活动轮廓与回顾状态 | 正式日报、复杂筛选、原始数据管理 |
| 今日回顾 | 材料筛选、事实核对与正式确认 | 项目管理、全文编辑器、复杂内容配置 |
| 历史回顾 | 按日期阅读已确认的每日回顾 | 周报、趋势、项目进度分析 |
| 数据中心 | 查看完整活动线索与进行针对性审查 | 解释今天主要做了什么 |
| 运行中心 | 提供记录状态信任、停止与恢复能力 | 默认展示全部内部运行细节 |
| 应用配置 | 单个应用的图标、分类和统计参与设置 | 批量运营、应用分析仪表盘 |
| 设置 | 低频全局偏好、隐私规则、回顾偏好与工作项目 | 每日主流程 |

### 3.3 今日总览信息层级

第一层：

- 最近活动与可识别线索
- 今日回顾状态
- 记录是否正常

第二层：

- 主要应用
- 活动时间分布
- 当前主要活跃时段

第三层：

- 记录数量
- 分类概览
- 剪贴板、浏览器等来源数量

第三层内容只在确实帮助理解时出现，不能为了填满卡片而保留。

### 3.4 今日回顾信息层级

#### 材料阶段

1. 当前需要用户判断的材料
2. 材料标题或内容线索
3. 时间、来源和隐私状态
4. 筛选与搜索
5. 总数、预计字数等辅助统计

预计字数不再作为核心信息。

#### 草稿阶段

1. 事项名称
2. 工作项目，可为空
3. 粗略时间
4. 一至两句事实摘要
5. 相关活动线索
6. 应用与时间校准信息
7. 形成记录与内部信息

### 3.5 历史回顾信息层级

左侧日期列表：

- 日期与星期
- 一至两个主要事项
- 弱化的项目信息

右侧内容：

- 最新确认的每日回顾正文
- 按需查看活动线索
- 文档级编辑入口

版本、形成方式、材料数量和原始内容属于二级信息。

### 3.6 数据中心信息层级

默认优先展示：

- 整体记录状态
- 最近记录事件
- 按时间组织的活动线索

不同记录来源的详细内容保留入口，但正常情况下默认折叠。数据中心是证据浏览器，不是产品首页，也不是长期个人数据中心。

### 3.7 工作项目与回顾偏好

“工作项目”只提供稳定的项目语境：

- 项目名称
- 简短说明
- 常见别名
- 活跃或归档状态

“回顾偏好”描述：

- 每日回顾希望采用的基本结构
- 关注的主要内容
- 表达重点与简洁程度

两者不成为顶层模块。它们归入设置中的回顾相关区域，并从今日回顾提供快捷入口。

工作项目不包含：

- 任务
- 截止日期
- 负责人
- 优先级
- 里程碑
- 项目进度
- 看板与统计

### 3.8 全局信息可信层级

> 用户确认的事实 > 回顾草稿 > 活动线索 > 辅助统计 > 内部信息

这个层级同时决定视觉强弱和默认展示位置。

### 3.9 统一命名

| 当前或容易误解的名称 | 目标名称 |
| --- | --- |
| Daily Report / 本地工作日报 | daily_report / 个人每日回顾 |
| 日报工作台 | 今日回顾 |
| 日报生成 | 不作为页面名称 |
| 历史日报 | 历史回顾 |
| 配置 Prompt | 回顾偏好 |
| 提交 | 确认 |
| 生成结果 | 回顾草稿 |
| 正式日报 | 每日回顾 |
| 隐藏 | 标记为敏感，或隐藏内容 |
| 排除 | 不用于本次回顾 |

### 3.10 Information Architecture 设计理由与边界

导航按照每日核心、运行查阅和管理分层，保证最常用的三项体验最容易找到，同时保留已有数据、配置和运行能力。它避免所有模块同等重要，也避免 daily_report 变成另一个数据中心、项目管理工具或知识库。

## 4. Interaction Design

### 4.1 全局交互原则

1. 每个操作范围最多只有一个主操作
2. 状态不伪装成按钮
3. 可逆操作优先直接执行并允许撤回
4. 不可逆且会造成真实损失的操作才二次确认
5. 详情按需展开，保持当前阅读位置稳定
6. 高频操作就地完成，低频管理进入二级层级
7. 不要求用户理解材料是如何被整理的

### 4.2 今日总览

#### 点击“开始今日回顾”

- 进入当天材料筛选阶段
- 不先进入回顾偏好
- 不要求确认日期

#### 点击“继续今日回顾”

- 回到当天离开时的阶段
- 保留用户已经完成的材料判断和草稿修改

#### 点击最近活动

- 先就地展开简要线索
- 用户选择“查看完整记录”后进入数据中心的对应上下文

#### 点击某个应用

- 查看该应用当天出现的主要时段和相关线索
- 完整管理仍进入数据中心或应用配置

#### 记录状态

- 正常时只显示小型状态与最近确认时间
- 需要注意时进入运行中心
- 严重异常时跨页面显示紧急停止入口

### 4.3 材料筛选

每条待确认材料提供：

- 用于本次回顾
- 不用于本次回顾
- 稍后判断
- 标记为敏感

语义说明：

- “不用于本次回顾”只影响当前回顾，不删除原始材料
- “稍后判断”继续保持不使用状态
- “标记为敏感”同时保护内容并从本次回顾排除
- “删除原始材料”是独立、低频、不可逆操作

材料选择不预设某个答案为正确。完成决定后，操作变为清晰状态并允许修改。

### 4.4 隐私保护

疑似敏感材料折叠时只显示时间、来源、判断原因和保护状态。内容由用户主动展开。

标记敏感后：

- 内容重新隐藏
- 从回顾中排除
- 提供短暂撤回
- 非阻塞建议“下次遇到相似材料时先让我确认”

相似规则使用人类可理解的描述。默认只让未来相似材料进入待确认。只有用户明确选择“始终视为敏感”，才建立稳定规则。

敏感材料在数据中心：

- 保留原时间位置
- 默认隐藏标题、地址与内容片段
- 搜索可以找到，但结果片段仍隐藏
- 专门审查时可以临时显示
- 离开后恢复隐藏
- 导出默认排除

### 4.5 回顾草稿

每个事项默认包含：

- 事项名称
- 工作项目，可为空
- 粗略时间，可为空
- 一至两句事实摘要
- 相关线索数量或查看入口

事项按照解释主要时间投入的能力排序，而不是按时间顺序、项目或价值排序。

同一语义事项即使跨越上午和下午，也可以合并，并保留轻量时段提示。相同应用中的不同目标应拆分为不同事项。

### 4.6 事实修正

#### 项目归属

- 点击项目名称即可改为其他工作项目或未归属
- 修改只影响当前事项
- 多次出现相同修正模式后，才建议将某个名称加入项目别名
- 项目别名只有经用户确认后才更新

#### 内容不准确

提供：

- 查看相关线索
- 修改内容
- 这件事今天没有发生
- 由我确认

无线索不等于虚构。用户记得确实发生过，可以作为本人确认的事实保留。用户无法确认且没有线索支持的内容必须修正或移除。

#### 时间修正

使用事件范围：

- 约 2 小时
- 约 3 至 4 小时
- 不足 15 分钟
- 暂不填写

事项时间与应用时间不要求精确对齐。

### 4.7 正式确认

- “确认每日回顾”不再出现第二个确认窗口
- 确认后停留在正文页面
- 不出现分数、庆祝、连续记录或明日计划
- 编辑已确认记录时，先恢复草稿身份，再重新确认
- 删除已确认记录需要说明影响并二次确认

### 4.8 历史回顾

- 默认打开最近一份已确认记录
- 左侧日期列表用于快速扫描近期主要事项
- 上一天、下一天与日期选择承担主要导航
- 点击事项按需展开活动线索
- 搜索覆盖日期、事项名称、工作项目和正文关键词
- 不提供周报、趋势、热力图和项目进度视图

### 4.9 应用配置

用户的高频任务是为单个应用：

- 设置图标
- 修改分类
- 设置是否参与统计
- 设置是否记录标题等必要属性

默认采用稳定紧凑的应用列表，不采用随宽度在两列和三列之间跳动的大型 Card。用户通常逐个配置，不增加批量选择能力。

### 4.10 运行中心

正常时优先展示：

- 正常记录中
- 最近活动线索时间
- 是否存在需要处理的问题

详细来源信息默认折叠。异常时只展开受影响部分。

状态统一为：

- 正常记录中
- 已停止
- 需要注意
- 状态未知
- 可能影响电脑使用

严重异常时，“紧急停止全部记录”成为唯一主操作，不二次确认。停止后明确展示实际结果，再提供“尝试恢复记录”“暂时保持停止”和“查看运行详情”。只有重新出现可信活动线索后，状态才恢复为正常。

### 4.11 Empty State

| 场景 | 建议表达 |
| --- | --- |
| 今日没有材料 | 今天暂时没有可用于回顾的活动线索 |
| 筛选无结果 | 没有符合当前条件的记录 |
| 没有待确认材料 | 没有需要你判断的材料，可以继续形成回顾草稿 |
| 没有历史回顾 | 还没有已确认的每日回顾 |
| 没有工作项目 | 尚未添加工作项目，事项仍可保持未归属 |
| 材料不足 | 现有线索不足以形成可信的回顾草稿 |

空状态不使用大型插图填充，也不暗示用户什么都没做。

### 4.12 Warning 与 Confirmation

状态反馈回答三个问题：

1. 发生了什么
2. 是否影响今日回顾
3. 用户是否需要行动

警告示例：

> 2 条材料可能包含敏感信息。它们暂时不会用于回顾草稿。

> 最近一段时间没有新的活动线索，今天的回顾可能不完整。

需要二次确认：

- 删除原始材料
- 删除已确认的每日回顾
- 删除隐私规则
- 放弃尚未保存的重要修改

不需要二次确认：

- 确认每日回顾
- 标记敏感
- 排除或恢复材料
- 紧急停止全部记录
- 修改项目归属

### 4.13 Interaction Design 设计理由与边界

交互优先帮助用户快速判断、少量修正、可信确认和顺利离开。它避免将主流程变成逐条标注、复杂文字编辑、Prompt 管理或系统运维流程。

## 5. Design System

### 5.1 设计语言

daily_report 的设计语言应体现：

- 安静
- 可信
- 克制
- 私密
- 清晰
- 不审判

### 5.2 Typography

- 全局使用同一现代无衬线字体家族
- 页面标题克制，不使用营销式大字号
- 事项标题使用中等字重
- 正文使用常规字重与舒适行距
- 元信息更小、更弱，但保持可读
- 日报与历史正文采用更宽松的阅读节奏
- 数据中心与应用配置采用紧凑排布
- 仅原始文件名、地址等必要内容使用等宽字体
- 信息层级主要通过字号与间距表达，不依赖大量加粗

### 5.3 Spacing

建议采用稳定间距序列：

- 4 至 8：图标、标签、同组元信息
- 12 至 16：组件内部、标题与说明
- 24：同一任务中的信息组
- 32 至 40：不同页面分区
- 48：阅读型页面的重要章节

越相关的信息越靠近，越独立的信息越分开。

### 5.4 信息密度

#### 阅读密度：宽松

适用于回顾草稿、已确认每日回顾和历史正文。

#### 工作密度：适中

适用于今日总览、今日回顾的主要操作区域、设置与异常处理。

#### 审阅密度：紧凑

适用于待确认材料、数据中心、应用配置列表和正常运行详情。

密度由页面任务固定决定，不提供全局紧凑或宽松设置。

### 5.5 Card 与页面表面

页面表面分为：

1. 页面背景
2. 内容分区
3. 独立信息单元
4. 临时浮层

Card 只用于内容具有明确边界、能够独立理解、需要单独比较或操作的情况。

Card 不用于：

- 填满布局
- 包裹一行状态
- 每条普通列表记录
- 日报正文的每个段落
- Card 内继续嵌套 Card

合理空白出现在内容之间和文档周围。不合理空白出现在有边界容器内部，暗示内容缺失。

### 5.6 Button 层级

#### 主操作

- 开始今日回顾
- 继续今日回顾
- 形成回顾草稿
- 确认每日回顾
- 重新确认

#### 次要操作

- 查看全部材料
- 调整回顾偏好
- 编辑事项
- 复制或导出每日回顾

#### 局部操作

- 查看线索
- 修改项目归属
- 展开详情
- 恢复默认值

#### 危险操作

- 删除原始材料
- 删除每日回顾
- 严重异常下的紧急停止

每个操作范围最多只有一个主操作。已保存、记录正常、已确认等状态不表现为按钮。

### 5.7 Corner Radius、Border 与 Shadow

- 4：标签、小型状态、紧凑信息块
- 6：按钮、输入和局部操作
- 8：Card、独立面板和浮层

普通内容主要依靠留白和细分隔线。完整边框只表达真实边界。普通 Card 不使用明显阴影，阴影只属于菜单、浮层和覆盖当前内容的临时区域。

选择状态使用细蓝色标记或淡背景，不同时叠加蓝色边框、阴影和高亮文字。

### 5.8 Color Hierarchy

约 85% 的界面由中性色构成，语义色小面积使用：

- 蓝色：当前选择、主操作、链接、当前流程阶段
- 绿色：记录正常、已保存、已确认
- 琥珀色：待判断、不确定、可能影响完整性
- 红色：删除、不可逆损失、严重异常与紧急操作
- 低饱和紫灰：敏感与隐私保护
- 中性灰：证据、原始材料、元信息与未归属状态

不能用绿色表示工作、红色表示娱乐。颜色不评价时间价值。

### 5.9 Light 与 Dark Theme

默认跟随系统，允许用户手动选择浅色或深色并保持选择。

浅色模式：

- 轻微冷灰页面背景
- 白色主要内容表面
- 深炭灰主要文字
- 低对比灰边框

深色模式：

- 中性炭黑背景，不使用纯黑或深蓝底
- 略亮的深灰内容表面
- 柔和灰白主要文字
- 依靠明度差分层，不使用发光边框

两种主题保持相同信息层级、状态含义和操作优先级。

### 5.10 Motion

- 100 至 160ms：按钮、选择与状态变化
- 180 至 240ms：局部展开与折叠
- 200 至 280ms：页面状态转换

Motion 只解释变化，不承担装饰作用。

不使用：

- 弹跳
- 卡片悬浮放大
- 持续闪烁或脉冲
- 图表每次打开重新增长
- 庆祝动画
- 虚假进度

材料决定后先在原位置显示新状态，再轻微移出待确认列表。查看线索保持当前阅读位置。确认每日回顾只改变文档身份。

### 5.11 State

状态层级：

- 中性状态：普通信息、主动停止、暂无内容
- 进行状态：正在整理、正在形成草稿
- 待判断状态：不确定材料或归属需要用户决定
- 完成状态：已保存、已确认、正常记录
- 保护状态：敏感内容受到保护
- 警告状态：可能影响回顾，需要检查
- 错误状态：操作未完成、可能丢失内容或无法继续

颜色不能单独承担含义，必须同时提供文字和图标。

### 5.12 Evidence、Draft 与 Final

#### Evidence

- 紧凑
- 可检查
- 中性色
- 时间和来源优先
- 不表现为已经确认的事实

#### Draft

- 文档感与审阅感并存
- 不确定内容明确标记
- 局部修正操作可见
- 相关线索按需查看

#### Final

- 稳定、宽松、适合阅读
- 只保留小型已确认标识
- 不散布编辑按钮
- 不显示应用排行榜、材料数量和形成过程

### 5.13 Design System 设计理由与边界

设计系统通过克制的中性色、稳定密度和明确状态，让用户相信内容、快速扫描并保持控制。它避免一页一套风格、Card 过度使用、蓝色装饰泛滥，以及生产力工具常见的压迫感。

## 6. Visual Style

### 6.1 风格定义

> daily_report 是以文档感为主、工具感为辅的安静个人回顾工作台。

它不是：

- 数据监控后台
- 传统工作日报系统
- 生产力评分工具
- 代码编辑器或终端
- 装饰性个人日记

### 6.2 视觉参考的取舍

| 参考 | 借鉴内容 | 不照搬内容 |
| --- | --- | --- |
| 现代个人博客 | 正文节奏、阅读宽度、段落留白 | 营销式首屏和大标题 |
| Raycast | 清晰操作层级、快捷而克制的工具感 | 过度依赖命令入口 |
| Linear | 状态表达、精确感、信息组织 | 团队项目管理心智 |
| Arc | 个人工具的亲近感 | 强装饰和品牌化控件 |
| JetBrains | 高密度管理页面的稳定性 | 开发工具术语和复杂面板 |
| Apple | 留白、克制、状态反馈 | 大面积展示性动画 |

### 6.3 页面视觉人格

- 今日总览：快速扫视、轻量克制，不追求填满
- 今日回顾：从紧凑审阅逐渐过渡到宽松文档
- 历史回顾：个人日记与可信档案的阅读感
- 数据中心：高密度证据索引，不是分析仪表盘
- 应用配置：稳定紧凑的工具列表
- 运行中心：正常时安静，异常时果断
- 设置：低频、直接，不增加解释性装饰

### 6.4 正式每日回顾的文档结构

默认结构：

1. 日期
2. 简短、事实性的今日概述
3. 按解释性权重排序的主要事项
4. 其他有意义的事项
5. 可选的结果、学习和未完成内容

每个主要事项包含：

- 事项名称
- 可选工作项目
- 粗略时间
- 一至两句事实摘要

不默认包含：

- 应用排行榜
- 活跃比例和空闲比例
- 生产力评价
- 原始搜索词、文件列表和网页流水
- 自动推测的问题原因
- 自动生成的明日计划
- 对工作时长的道歉或评价

### 6.5 Visual Style 设计理由与边界

文档感帮助用户把结果当作可信记录，工具感帮助用户高效审阅。两者结合能够形成私密、安静而精确的体验，同时避免产品看起来像复杂后台、监控系统或内容生成器。

## 7. Design Principles

### 7.1 先事实，后统计

用户首先理解做了什么，再查看时间、应用和来源。数字不能代替事项事实。

### 7.2 真实性优先于完整性

可以遗漏、留白或只有一个事项，但不能写入用户无法确认的内容。

### 7.3 隐私边界先于内容形成

高风险和不确定材料默认不使用。用户决定什么可以进入回顾。

### 7.4 自动整理，用户确认

产品负责降低整理成本，用户负责最终判断。未确认草稿不是长期可信记录。

### 7.5 不审判用户

工作、学习、娱乐和个人活动都按事实与时间解释价值处理，不进行效率评价。

### 7.6 支撑能力让位于核心体验

数据、配置、运行和内部信息必须让位于材料筛选、事实核对和正式确认。

### 7.7 每天负责，长期交给其他工具

daily_report 负责每一天的可信回顾，不承担跨周趋势、项目管理和知识管理。

### 7.8 允许快速离开

产品的成功不是增加停留时间，而是让用户快速获得清楚感并离开。

### 7.9 不为填满界面增加信息

空白可以是合理的。没有明显体验价值的统计、卡片和功能应删除或降级。

### 7.10 现有能力不等于同等重要

考虑当前产品已经存在较多能力，UX 允许保留现有模块，但需要重新排列优先级、默认状态和视觉权重。已实现不构成核心入口的理由。

## 附录 A：Review 结论

### A.1 下班前正常回顾

三阶段流程能够在约 8 至 20 分钟内完成。全量材料审查是可选路径，不会强迫用户逐条阅读。

### A.2 工作中临时查看

今日总览支持 30 秒至 2 分钟的被动查看，不触发回顾流程。日期范围降为次要查询，避免今日总览成为通用统计页。

### A.3 历史回看

日期列表展示一至两个主要事项，已经能够帮助用户理解最近在做什么，不需要额外近期汇总页。

### A.4 隐私材料审查

本次不使用、稍后判断、标记敏感和删除原始材料拥有不同语义。未来相似规则默认进入待确认，不自动扩大敏感范围。

### A.5 运行异常

正常状态压缩，严重异常跨页面提示并提供直接紧急停止。详细能力保留在二级层级。

### A.6 材料不足

产品允许一句话、一个事项或不创建记录。没有活动线索不等于用户没有活动。

### A.7 事实与项目修正

用户核对事项，不核对全部材料。项目修正默认只影响当前事项，多次重复后才建议增加项目别名。

### A.8 整体一致性

当前设计在以下关系上没有明显冲突：

- 被动查看与主动回顾
- 低负担与事实可信
- 工作项目语境与非项目管理边界
- 全天回顾与非评判性
- 隐私保护与证据追溯
- 保留现有能力与降低视觉优先级

## 8. Open Questions

以下问题不阻塞当前 UX 基线，但必须通过真实使用验证：

### 8.1 回顾成本

- 正常一天实际需要多少分钟
- 全量材料审查会被使用多频繁
- 哪一步最容易让用户中途离开

### 8.2 草稿质量

- 每天平均需要修正多少个事项
- 项目归属错误出现多频繁
- 无法确认或虚构内容的发生频率
- 默认表达是否仍然过于啰嗦

### 8.3 线索覆盖

- 回顾实际能够解释多少主要时间
- 哪类活动最容易遗漏
- 线下事项补充是否足够轻量
- 应用时间是否真的帮助校准事项时间

### 8.4 隐私负担

- 每天待判断材料数量是否可接受
- 用户主动进入全量审查的频率
- 相似材料建议是否准确、透明且不过度打扰
- 敏感材料默认隐藏是否足够安心

### 8.5 工作项目语境

- 项目名称、说明和别名是否足以改善归属
- 项目别名建议出现的频率是否合适
- 工作项目入口是否容易找到但又不抢占主流程

### 8.6 页面与主题

- 深色模式下长文本阅读是否舒适
- 浅色模式在明亮办公环境中是否清晰
- 今日总览减少 Card 后是否仍然容易扫描
- 应用配置列表是否解决两列和三列跳变造成的空荡感

### 8.7 运行信任

- 全局小型状态是否足以让用户放心
- 严重异常提示是否足够醒目但不过度制造压力
- 停止与恢复后的状态是否容易理解

- 核心页面的高保真交互原型

## 10. UX Design 完成后的下一步

当前完成的是 UX Design 基线，不代表页面已经经过真实使用验证。下一步仍然属于 UX Design，而不是技术设计。

### 第一步：建立高保真核心原型

优先覆盖五个核心页面状态：

1. 今日总览
2. 今日回顾：材料筛选
3. 今日回顾：回顾草稿
4. 已确认的每日回顾
5. 历史回顾

同时补充两个关键异常状态：

- 敏感材料审查
- 严重运行异常

原型必须使用接近真实的一天数据，不能使用只有两三条内容的理想示例。

### 第二步：完成真实场景走查

至少使用以下场景逐一演练：

- 普通碎片化工作日
- 单一事项的简单一天
- 包含工作、学习与娱乐的一天
- 存在敏感材料的一天
- 记录中途缺失的一天
- 草稿项目归属错误的一天
- 严重运行异常

每次走查记录：

- 完成时间
- 停顿和犹豫位置
- 不理解的名称
- 多余操作
- 错误修正数量
- 是否最终获得清楚感

### 第三步：连续真实试用

因为目标用户就是产品所有者本人，最有价值的验证不是招募泛化用户，而是连续使用若干真实工作日。

建议先验证 5 至 7 个不同类型的日子，观察：

- 是否愿意自然打开
- 是否能够在目标时间内完成
- 是否经常跳过材料审查
- 草稿是否大部分正确
- 是否真正缓解时间不确定感
- 哪些页面仍然像管理后台

### 第四步：Review 与迭代

根据真实试用，只修改反复出现的问题：

- 删除低价值信息
- 调整流程顺序
- 修改交互文案
- 调整信息密度
- 提升隐私判断与事实修正效率

不因一次偶发不便增加新模块。

### 第五步：形成页面级 UX Specification

原型验证通过后，再补充：

- 每个页面的目标与进入条件
- 默认状态、空状态、异常状态和完成状态
- 每个操作的结果与撤回方式
- 页面之间的进入与返回关系
- 最终内容文案
- 明暗主题中的视觉规则

完成这一轮 Review 并由用户明确确认后，UX Design 阶段才可以视为结束。之后是否进入其他阶段，需要重新确认；本文不包含后续技术或开发安排。

## 11. 最终体验验收标准

当以下条件成立时，daily_report UX Design 可以通过最终验收：

- 用户打开应用后无需思考就能理解今天的活动轮廓
- 用户清楚知道查看与主动回顾是两种不同状态
- 用户不需要逐条检查全部材料
- 隐私材料默认受到保护
- 草稿主要事项大部分真实，错误能够局部修正
- 项目归属修改快捷，但没有项目管理负担
- 粗略时间足以帮助理解，不制造分钟级精确幻觉
- 正式每日回顾覆盖工作、学习、娱乐和个人活动
- 没有生产力评价、压力或庆祝机制
- 正常回顾通常可以在 8 至 20 分钟内完成
- 材料不足时仍能形成极简但真实的记录
- 用户确认后能够自然结束并离开产品
- 历史记录可以帮助用户理解最近几天在做什么
- 运行异常时能够快速停止并理解当前状态
- 深色与浅色主题保持相同的信息层级和可读性
- 页面不依靠无意义 Card、统计和功能填满布局
