# Product Context

daily_report 是只为产品所有者本人服务的个人每日数字活动回顾工具。

它不是：

- 工作日报生成器
- 精确时间追踪器
- 任务管理工具
- 项目管理工具
- 长期知识管理系统
- 效率评分工具
- 员工监控工具
- 个人原始数据中心

它的核心目标是：

> 在不要求用户白天持续手工记录的情况下，帮助用户在下班前用十几分钟到半小时，把当天分散的数字行为线索整理成经过用户确认的事实性每日回顾，从而解释今天主要时间去了哪里。

产品成功不是让用户停留更久，而是让用户更快获得清楚感并离开。

# Core UX Baseline

以下决策已经确认，Prototype 必须遵守。

## 使用场景

- 默认打开进入“今日总览”
- 核心使用时刻是下班前或晚间
- 工作中临时查看是一条被动短流程
- 用户偶尔按日期回看最近几天
- 当天回顾可以不完整，但必须真实
- 用户可以选择某一天不创建每日回顾

## 核心流程

主动回顾采用三个阶段：

1. 材料筛选
2. 回顾草稿
3. 正式确认

“回顾偏好”不属于每日必经步骤。

筛选材料、形成草稿和确认每日回顾都必须是用户主动行为。

## 信息优先级

> 用户确认的事实 > 回顾草稿 > 活动线索 > 辅助统计 > 内部信息

用户首先看“做了什么”，时间与应用统计其次。

## 每日回顾内容

- 覆盖工作、学习、娱乐和个人活动
- 按解释主要时间投入的能力排序
- 项目归属可以为空
- 时间使用粗略范围，不追求分钟精度
- 结果、学习和未完成内容都是可选的
- 应用排行榜与统计图表不进入正式正文
- 不自动生成明日计划
- 不评价工作时长和生产力

## 材料与隐私

每条待确认材料的最终措辞是：

- 用于本次回顾
- 不用于本次回顾
- 稍后判断
- 标记为敏感

尚未判断的材料默认不使用，但不阻塞形成草稿。

“不用于本次回顾”“标记为敏感”和“删除原始材料”是不同语义。

全量材料审查入口必须保留，但不是必经步骤。

## Evidence

- Evidence 是回忆线索，不是绝对证明
- 点击回顾事项可以按需查看相关来源、时间与材料标题
- 被排除的敏感材料不能通过 Evidence 重新出现
- 用户记忆可以确认没有数字线索支持的真实事项
- 用户无法确认且没有线索支持的内容必须修正或移除

## 工作项目

工作项目只提供归属语境：

- 名称
- 简短说明
- 常见别名
- 活跃或归档状态

不增加任务、进度、截止日期、优先级、里程碑或看板。

项目修正默认只影响当前事项。多次出现相同模式后，才建议增加项目别名。

## 历史回顾

- 只收录已确认的每日回顾
- 默认打开最近一份确认记录
- 日期列表显示一至两个主要事项
- 不增加周报、趋势、热力图或项目统计

## 运行状态

- 正常状态保持安静，只显示整体状态与最近记录时间
- 详细来源信息正常时默认折叠
- 严重异常可以跨页面提示
- “紧急停止全部记录”在严重异常时成为唯一主操作
- 已停止是中性状态，不等于异常

## 视觉方向

> 以文档感为主、工具感为辅的安静个人回顾工作台。

- 深色主题是主要使用环境，浅色主题适配明亮办公环境
- 中性色占绝大部分
- 蓝色只用于操作与选择
- 普通页面少 Card、少阴影、少装饰
- 阅读、工作、审阅采用不同固定密度
- Evidence、Draft、Final 必须拥有不同视觉身份
- 不使用庆祝动画、分数、连续记录和生产力颜色

# Existing Product Constraints

daily_report 已经存在较多页面和能力，这次设计属于 UX 补课，不是假设从零开始。

现有主要区域包括：

- 今日总览
- 数据中心
- 日报工作台
- 应用配置
- 运行中心
- 设置

现有能力大部分可以保留，但不代表它们需要保持同等视觉权重。

Prototype 应采用以下原则：

- 保留当前仍有明确用户价值的能力
- 优先调整名称、顺序、默认状态和信息层级
- 不为了理想结构轻易删除用户已经依赖的入口
- 不为了迁就现有布局继续保留低价值 Card 和统计
- 当理想体验与现有行为冲突时，明确记录“目标体验”和“可接受妥协”
- 不讨论妥协如何实现

已确认的目标导航层级是：

**每日核心**

- 今日总览
- 今日回顾
- 历史回顾

**运行与查阅**

- 数据中心
- 运行中心

**管理**

- 应用配置
- 设置

## Incremental Redesign Guardrail

本阶段不是从零设计一款新产品，也不是默认进行全量重设计。

现有 daily_report 是 Prototype 的起点。除非真实场景走查证明某部分明显阻碍核心回顾，否则默认采用渐进式改进。

对每个页面和能力，必须先归入以下一种处置：

1. **保持现状**：当前体验已经顺手，不为了视觉统一强行修改
2. **保留并局部调整**：保留页面职责和主要操作，只调整名称、顺序、比例、密度、默认展开状态或视觉权重
3. **结构性调整**：当前结构持续阻碍核心任务，经过对比验证和用户明确确认后才调整

未经用户单独确认，不得：

- 提出推倒整个现有导航或页面体系
- 删除当前已经依赖的主要能力
- 将所有页面统一替换成新的布局模式
- 为追求视觉一致性重画所有页面
- 把局部布局问题扩大为全产品重构
- 假设现有页面没有任何保留价值

当前页面的默认处置是：

| 页面 | 默认处置 |
| --- | --- |
| 今日总览 | 保留主要内容，调整信息优先级、Card 数量和状态表达 |
| 数据中心 | 保留整体结构，强化最近记录与隐私语义，详细来源默认折叠 |
| 今日回顾 / 原日报工作台 | 允许重点调整，因为当前步骤结构与用户心智存在明显冲突 |
| 历史回顾 | 保留左右结构，正文优先，版本与形成信息降级 |
| 应用配置 | 保留单应用配置能力，将不稳定卡片网格调整为稳定列表关系 |
| 运行中心 | 保留现有安全与管理能力，重新分层正常、异常和紧急操作 |
| 设置 | 保留现有入口和职责，只整理回顾偏好、工作项目与隐私规则的层级 |

每次提出明显变化时，必须同时展示：

- 当前体验保留什么
- 当前问题是什么
- 最小调整方案是什么
- 为什么最小调整仍然不足，才需要更大调整
- 变化会影响哪些已有使用习惯

优先展示“现有结构 → 局部调整后结构”的对比。只有局部调整无法解决已验证的问题时，才讨论结构性调整。

结构性调整不能在同一轮自动视为确认。Agent 必须先提出理由和对比，等待用户明确接受。

# Prototype Workflow

整个阶段遵循：

> Prototype → Review → Iterate

不要一次性画完所有页面。按照以下顺序推进。

## Step 1：建立真实的一天样本

第一轮不要直接画页面。

先与我共同确定一个接近真实的“一天”，至少包含：

- 两至三个工作项目
- 一个会议或集中事件
- 一段文档或开发工作
- 零散浏览和资料调研
- 学习内容
- 娱乐或个人活动
- 一条疑似敏感材料
- 一个项目归属错误风险
- 一段没有活动线索的时间
- 一个用户记得但没有数字材料的事项

这个样本将贯穿全部原型，保证不同页面展示的是同一天、同一组事实。

本轮输出：

- 当天背景
- 关键事件清单
- 活动线索摘要
- 敏感与不确定材料
- 预期回顾草稿
- 用户最终希望确认的每日回顾

## Step 2：定义 Prototype Scope 与 State Matrix

明确原型必须覆盖哪些页面和状态。

核心页面：

1. 今日总览
2. 今日回顾：材料筛选
3. 今日回顾：回顾草稿
4. 已确认的每日回顾
5. 历史回顾

重要支撑页面或状态：

- 全量材料审查
- Data Center 中的敏感材料
- 工作项目与回顾偏好
- 应用配置列表
- 运行中心正常状态
- 严重运行异常

必须覆盖的状态：

- 默认状态
- 内容较多
- 内容较少
- 空状态
- 待判断状态
- 敏感保护状态
- 草稿状态
- 已确认状态
- 跨天未完成
- 状态未知
- 严重异常

本轮输出一份页面与状态矩阵，并明确哪些进入高保真、哪些只做结构验证。

## Step 3：核心流程结构原型

先形成内容真实的中等保真结构，不先追求视觉细节。

按照完整流程讨论：

> 今日总览 → 材料筛选 → 回顾草稿 → 正式确认

每个页面需要标明：

- 页面目标
- 第一视觉焦点
- 信息顺序
- 唯一主操作
- 次要操作
- 进入与离开方式
- 当前状态
- 内容较多时的表现
- 内容不足时的表现

使用同一天样本验证内容是否能够自然放入页面。

## Step 4：核心流程高保真原型

结构确认后，再应用已经确定的视觉系统：

- Typography
- Spacing
- 阅读、工作与审阅密度
- Surface 与 Card 规则
- Button 层级
- Color Hierarchy
- Corner Radius、Border 与 Shadow
- Motion 与状态反馈

优先完成深色主题，再验证浅色主题。浅色与深色必须保持相同的信息优先级。

## Step 5：次要路径原型

依次验证：

1. 工作中临时查看
2. 历史回看
3. 全量材料审查
4. 敏感材料后续查看
5. 单个应用配置
6. 工作项目与回顾偏好

次要路径不能反向增加核心回顾流程的复杂度。

## Step 6：安全与异常原型

重点验证：

- 材料不足但用户仍想回顾
- 只有一个事项
- 用户选择不创建某一天的回顾
- 昨日草稿未确认
- 敏感材料误判与撤回
- 无法确认的草稿事项
- 项目归属错误
- 记录状态未知
- 严重异常与紧急停止
- 恢复后的状态确认

异常状态必须说明：

- 发生了什么
- 是否影响今日回顾
- 用户现在可以做什么

## Step 7：Scenario Review

使用原型逐一完成：

- 普通碎片化工作日
- 单一事项的简单一天
- 同时包含工作、学习和娱乐的一天
- 有敏感材料的一天
- 记录中途缺失的一天
- 草稿项目归属错误的一天
- 严重运行异常

每次 Review 记录：

- 用户目标是否清楚
- 第一眼是否看到正确内容
- 下一步是否自然
- 是否出现多余操作
- 是否出现无法解释的状态
- 是否需要逐条阅读过多材料
- 是否能够顺利离开
- 哪些内容应该删除或降级

## Step 8：Iterate 与收敛

只修改反复出现的问题：

- 信息层级错误
- 流程顺序错误
- 操作含义不清
- 页面密度不合适
- 隐私判断成本过高
- 事实修正不够直接
- 视觉状态容易误解

不要因为一次偶发需求新增模块。

## Step 9：形成 UX Prototype Specification

最终文档至少包括：

1. Prototype Objective
2. Realistic Day Scenario
3. Prototype Scope
4. Page and State Matrix
5. Core Flow Prototype
6. Secondary Flow Prototype
7. Exception and Safety States
8. Interaction Annotations
9. Light and Dark Visual Direction
10. Scenario Review Findings
11. Iteration Decisions
12. Remaining Open Questions
13. UX Acceptance Checklist

每一项都说明：

- 设计决策是什么
- 为什么这样设计
- 如何服务 daily_report 核心价值
- 避免了哪些产品膨胀

# Discussion Rules

整个过程每轮只讨论一个主题。

每轮开始时：

- 说明本轮要验证什么
- 说明它与核心流程的关系
- 给出你的明确推荐，而不是只提供选项

每轮结束时总结：

- 目前已经确定什么
- 原型暴露了什么问题
- 仍然存在什么问题
- 哪些问题延后讨论
- 下一轮验证什么

不要一次提出多个宽泛问题。

优先提出一个能够推进设计的关键问题。没有必要询问时，基于现有 UX 文档直接给出推荐。

如果发现某个设计只是增加功能、统计、Card 或操作，但没有帮助用户更快、更安心、更可信地完成回顾，请明确建议删除或降级。

# Visual Communication

用户仅阅读长段文字时很难判断空间关系，因此：

- 流程、结构对比和页面层级应尽量使用可视化表达
- 需要比较方案时，展示并列结构，而不是只描述抽象差异
- 原型应使用真实长度的中文内容，避免过短占位文字
- 同一轮只展示当前讨论范围，不一次性堆叠所有页面
- 视觉原型必须同时考虑宽内容、长标题、材料数量多和侧边导航展开的情况
- 内容不足时允许自然留白，不用低价值卡片填满页面
- 不展示代码或实现方式

每次展示原型后，解释用户应该重点观察什么，以及当前原型在验证哪项 UX 假设。

# Prototype Acceptance Criteria

Prototype 阶段完成时必须满足：

- 用户打开应用后能立即理解今天的活动轮廓
- 用户能清楚区分被动查看与主动回顾
- 核心流程只有材料筛选、回顾草稿、正式确认三个阶段
- 用户不需要逐条阅读全部材料
- 全量材料审查入口仍然存在
- 敏感材料默认受到保护
- 草稿首先突出“做了什么”
- 项目归属能够局部快速修正
- 用户能够按需查看 Evidence
- 无法确认的事实不会悄悄进入正式记录
- 应用时间只用于辅助校准
- 一句话或一个事项也可以形成正式回顾
- 用户可以明确跳过某一天
- 历史回顾适合阅读，不是统计后台
- 正常运行状态安静，严重异常能够快速停止
- 深色与浅色保持相同层级
- 页面不依靠无意义 Card 和统计填充布局
- 正常回顾在模拟中能够控制在 8 至 20 分钟
- 用户确认后能够自然结束并离开

---

# Output Requirement

最终输出：

> daily_report UX Prototype Specification

不要输出：

- 技术方案
- 开发任务拆分
- 代码
- 数据库设计
- API 设计
- 组件实现方式
- 性能方案
- 工期估算

当前阶段完成后，只能判断 UX Prototype 是否通过 Review。是否进入其他阶段，必须由用户重新明确确认。

---

# First Round Instruction

现在开始第一轮：

> 建立贯穿整个 Prototype 的“真实一天样本”。

请先阅读两份上游文档，然后：

1. 总结这个样本必须覆盖的体验风险
2. 根据已知用户行为，提出一版真实一天样本草案
3. 区分活动事实、原始线索、敏感材料、缺失时段和用户记忆补充
4. 提出一个最需要用户确认的问题

本轮不要开始画完整页面，也不要讨论实现。

---

# Important Reminder

daily_report 的核心不是形成一份看起来完整的报告。

而是让用户以可接受的成本确认一份真实但允许不完整的每日事实，从而理解主要时间去了哪里。

任何原型设计如果不能帮助用户：

- 更快理解今天
- 更安心控制隐私
- 更容易修正事实
- 更可信地完成确认
- 更自然地离开产品

就应该被简化、降级或删除。
